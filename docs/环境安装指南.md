# 环境安装与运行指南

> 适用版本：当前仓库
> 更新日期：2026-07-10

## 一、推荐方式：Compose 启动完整系统

当前仓库的统一入口是根目录 `compose.yaml`。Compose 会启动：

- PostgreSQL 15 + PostGIS 3.4；
- Spring Boot 后端；
- OSRM 路由服务；
- Nginx 前端与同源反向代理。

数据库表由 Flyway 自动迁移，无需手工执行 `db/migration` 中的任何 SQL。

### 1.1 前置条件

安装下列任一组合：

- Podman 4+ 与 `podman compose` / `podman-compose`；
- Docker 24+ 与 Docker Compose v2。

检查：

```bash
podman --version
podman compose version
```

或：

```bash
docker --version
docker compose version
```

首次构建还需要可访问 Maven Central、npm registry、容器镜像仓库和 Geofabrik 下载站。

### 1.2 准备环境变量

从仓库根目录执行：

```bash
cp .env.example .env
```

`.env.example` 的值只用于本机课程演示。若服务要暴露到局域网或公网，至少修改：

- `POSTGRES_PASSWORD`；
- `JWT_SECRET`，建议使用 64 字符以上随机值；
- `BIND_ADDRESS`，保持 `127.0.0.1` 可避免意外对外暴露。
- `CORS_ALLOWED_ORIGINS`；若从其他主机名或 IP 打开页面，必须写入实际的
  `http(s)://主机:端口`，否则 WebSocket Origin 校验会拒绝连接。
- `UPLOAD_MAX_IMAGE_BYTES` 和 `UPLOAD_MAX_IMAGE_PIXELS`，默认为 5 MB 与 2500 万像素，用于限制 PNG/JPEG 上传。

不要提交包含真实密钥的 `.env`。

### 1.3 一键启动

```bash
./deploy/up.sh
```

脚本会按顺序选择 `podman compose`、`podman-compose` 或 `docker compose`，然后执行构建与后台启动。

也可直接运行：

```bash
podman compose up -d --build
```

或：

```bash
docker compose up -d --build
```

### 1.4 服务地址

默认仅绑定本机：

| 服务 | 地址 |
|---|---|
| 前端统一入口 | `http://127.0.0.1:8088` |
| 移动公众上报页 | `http://127.0.0.1:8088/report` |
| 后端调试端口 | `http://127.0.0.1:8080` |
| OpenAPI | `http://127.0.0.1:8080/v3/api-docs` |
| Swagger UI | `http://127.0.0.1:8080/swagger-ui.html` |
| OSRM 直连 | `http://127.0.0.1:5000` |
| PostGIS | `127.0.0.1:5432` |

Nginx 在前端入口提供同源代理：

- `/api/**` → 后端；
- `/ws/**` → 后端 WebSocket；
- `/uploads/**` → 后端上传文件；
- `/osrm/**` → OSRM。

两个匿名写入路径有独立的单 IP 限流：`POST /api/reports` 平均 6 次/分钟（突发 5），`POST /api/reports/{trackingCode}/images` 平均 20 次/分钟（突发 10）。超限返回 HTTP 429。

## 二、首次启动说明

### 2.1 数据库

PostGIS 首次健康后，后端启动并自动执行：

- `backend/src/main/resources/db/migration/V1__init.sql`；
- `backend/src/main/resources/db/migration/V2__complete_business_workflows.sql`。

Flyway 会记录已执行版本。迁移一旦用于持久化数据卷，不要改写旧文件；新增变更应创建下一版本迁移。

### 2.2 OSRM

默认下载江苏 OSM PBF，并执行：

1. `osrm-extract`；
2. `osrm-partition`；
3. `osrm-customize`；
4. `osrm-routed`。

首次下载和 MLD 预处理可能需要数分钟到数十分钟。此时 `osrm` 显示 `starting` 属于正常现象，处理结果保存在 `osrm_data` 命名卷，后续启动会复用。

查看进度：

```bash
podman compose logs -f osrm
```

Docker 用户替换为：

```bash
docker compose logs -f osrm
```

## 三、启动后检查

### 3.1 容器状态

```bash
podman compose ps
```

或：

```bash
docker compose ps
```

PostGIS、后端和前端应最终为健康状态；OSRM 在首次预处理期间可能保持 `starting`。

### 3.2 后端健康

```bash
curl --fail http://127.0.0.1:8080/actuator/health
```

预期：

```json
{"status":"UP"}
```

### 3.3 前端兼容接口

```bash
curl --fail http://127.0.0.1:8088/api/resources
curl --fail http://127.0.0.1:8088/api/stations
```

应分别返回资源数组与泵车数组。

### 3.4 空间查询

```bash
curl --fail \
  -H 'Content-Type: application/json' \
  -d '{"name":"","type":"waterlogging"}' \
  http://127.0.0.1:8088/api/spatial-query
```

预期返回 GeoJSON `FeatureCollection`。

### 3.5 OSRM 路由

等待 OSRM 健康后执行：

```bash
curl --fail \
  'http://127.0.0.1:8088/osrm/route/v1/driving/117.1855,34.2815;117.2266,34.2522?overview=full&geometries=geojson'
```

预期响应中的 `code` 为 `Ok`。

## 四、演示账号

Flyway 种子数据提供以下本地演示账号：

| 用户名 | 初始密码 | 角色 |
|---|---|---|
| `admin` | `admin123` | 管理员 |
| `operator` | `admin123` | 操作员 |
| `viewer` | `admin123` | 查看者 |

这些账号只用于课程演示。共享部署前应修改或删除默认账号。

登录示例：

```bash
curl --fail \
  -H 'Content-Type: application/json' \
  -d '{"username":"admin","password":"admin123"}' \
  http://127.0.0.1:8080/api/auth/login
```

不要把返回 Token 写入仓库、日志或报告。受保护接口使用：

```http
Authorization: Bearer <token>
```

## 五、本地开发方式

完整联调优先使用 Compose。仅开发单个模块时可按以下方式启动。

### 5.1 数据库

只启动 PostGIS：

```bash
podman compose up -d postgis
```

或：

```bash
docker compose up -d postgis
```

`application.yml` 的本地默认连接为：

- 数据库：`flood_gis`；
- 用户：`gis`；
- 密码：`gis123`；
- 地址：`jdbc:postgresql://localhost:5432/flood_gis`。

若 `.env` 使用了不同密码，启动原生后端时同步设置 `DATABASE_PASSWORD` 等环境变量。

### 5.2 后端

前置条件：JDK 17。仓库同时提供 Linux/macOS 与 Windows 的 Maven Wrapper，
无需预先安装系统 Maven。

从根目录运行：

```bash
cd backend
./mvnw spring-boot:run
```

测试：

```bash
cd backend
./mvnw clean verify
```

Windows 在 `backend` 目录运行 `mvnw.cmd spring-boot:run` 或
`mvnw.cmd clean verify`。

### 5.3 前端

前置条件：Node.js 22 和 npm。

```bash
cd frontend
npm ci
npm run dev
```

生产构建：

```bash
npm run build
```

`npm run build` 同时执行 Vue TypeScript 检查和 Vite 构建。

### 5.4 当前前端联调限制

现有 `frontend/src/main.ts` 无条件加载 `frontend/src/mock/api.ts`，会拦截资源、泵车和空间查询；工单与公众上报也仍保存在 Pinia/`localStorage`。因此：

- `npm run dev` 适合查看现有 Mock 演示；
- 启动真实后端不会自动使页面进入真实数据模式；
- 真实联调需先按 `docs/前端修改建议.md` 增加 Mock 开关并接入认证、工单、上报和 WebSocket；
- 本轮遵守前端零修改约束，尚未处理该限制。

## 六、常用运维命令

以下以 Podman 为例；Docker 用户替换命令前缀。

```bash
# 查看状态
podman compose ps

# 查看最近日志
podman compose logs --tail=200

# 持续查看后端日志
podman compose logs -f backend

# 重新构建单个服务
podman compose up -d --build backend

# 停止并保留数据
podman compose down
```

删除所有数据卷：

```bash
podman compose down -v
```

`down -v` 会永久删除业务数据库、上传图片和 OSRM 预处理缓存，执行前必须确认。

## 七、自定义 OSRM 数据

使用裁剪后的徐州 PBF 时，在 `.env` 中设置：

```dotenv
OSRM_DATA_URL=https://example.invalid/xuzhou.osm.pbf
OSRM_DATA_FILE=xuzhou.osm.pbf
OSRM_PROFILE=car
OSRM_FORCE_REBUILD=true
```

首次成功后将 `OSRM_FORCE_REBUILD` 改回 `false`。可用 profile 为 `car`、`bicycle`、`foot`。

对外展示 OSM 数据时应保留 OpenStreetMap/ODbL 署名。

## 八、故障排查

### 8.1 PostGIS 不健康

- 检查 `POSTGRES_PORT` 是否被占用；
- 查看 `postgis` 日志；
- 确认 `POSTGRES_DB`、`POSTGRES_USER`、`POSTGRES_PASSWORD` 一致；
- 已存在的数据卷不会因修改环境变量自动重建用户密码。

### 8.2 后端不健康

- 先确认 PostGIS 健康；
- 查看 Flyway 是否有迁移失败；
- 检查 `JWT_SECRET` 长度和数据库连接变量；
- 检查上传卷对 UID 10001 是否可写。

### 8.3 OSRM 长时间 Starting

- 查看下载与预处理日志；
- 确认网络可以访问 `OSRM_DATA_URL`；
- 检查磁盘空间和内存；
- 下载中断后可重启，脚本会移除未完成的 `.part` 文件并重试。

### 8.4 页面有数据但后端没有访问日志

这是当前前端 Mock 拦截的表现。检查 `frontend/src/main.ts` 和浏览器 Network 面板，处理方式见 `docs/前端修改建议.md` 的 P0-1。

### 8.5 上传后图片 404

- 检查后端 `UPLOAD_DIRECTORY` 与 Compose 上传卷；
- 确认返回 URL 以 `/uploads/` 开头；
- 通过后端调试端口直接请求图片，区分后端静态映射与 Nginx 代理问题；
- 查看文件是否实际写入 `uploads_data`。

如果返回 400，请确认文件为 PNG/JPEG、声明类型与实际格式一致，且未超过字节数或像素上限。如果返回 409，请确认追踪码对应 `REPORT` 尚处于 `PENDING/PROCESSING`，且已有图片少于 5 张。

## 九、当前验证边界

当前后端已在临时 PostgreSQL 16 + PostGIS 3.6 上通过 Flyway、健康检查、关键 REST/RBAC、空间查询、公众上报、派单、图层、图片和 STOMP 认证烟测。Compose 配置和脚本已通过静态检查。

尚需在具备常驻容器运行能力的机器补充：

- 从空卷完整冷启动四服务；
- OSRM 下载、MLD 预处理和路径响应；
- Nginx 同源 API/WS/上传/OSRM 代理；
- 容器重启后的数据库、上传和 OSRM 缓存持久化；
- 关闭 Mock 后的真实浏览器端到端联调。
