# 前端修改建议

> 审计日期：2026-07-10
> 审计范围：`frontend/**` 只读检查
> 约束：本轮未修改任何前端文件

## 结论

现有前端可以通过 TypeScript 检查和生产构建，也能作为独立 Mock 演示运行。当前首要问题是数据链路仍停留在 Mock、Pinia 和 `localStorage`：即使后端与 Nginx 已启动，主要业务操作也不会形成真实数据闭环。

建议先处理 P0 联调阻断，再修复 P1 的业务一致性与安全问题，最后完成 P2 的体验和工程质量优化。

| 优先级 | 数量 | 处理目标 |
|---|---:|---|
| P0 | 3 | 让真实后端、工单和公众上报进入主数据链路 |
| P1 | 4 | 修复调度正确性、交互冲突、XSS 与上传边界 |
| P2 | 4 | 修复引导、生命周期、项目元信息和包体积 |

## P0：联调阻断

### P0-1：Mock 无条件覆盖全局 Fetch，三个真实接口永远收不到请求

**位置**

- `frontend/src/main.ts:5`
- `frontend/src/mock/api.ts:245-315`
- 调用点：`frontend/src/store/emergency.ts:86-108`、`frontend/src/components/SpatialQueryPanel.vue:470-504`

**现象**

`main.ts` 在所有构建模式下执行 `import './mock/api'`。该模块重写 `window.fetch`，只要 URL 包含 `/api/resources`、`/api/stations` 或 `/api/spatial-query` 就直接返回本地数据。生产环境中的 Nginx 代理和真实后端无法接收到这些请求。

**影响**

- 后端资源状态、数据库空间设施和前端显示长期不一致；
- REST、权限、错误处理和 PostGIS 查询无法通过页面验证；
- 网络面板会呈现“请求成功”的假象，增加联调误判风险。

**建议**

1. 将 Mock 改为显式构建开关，例如 `VITE_USE_MOCK=true` 时才动态加载；
2. 生产构建和容器镜像固定关闭 Mock；
3. 使用 MSW 或独立开发服务器提供 Mock，避免改写全局 `window.fetch`；
4. 为真实请求建立统一 API 客户端，并集中处理 `Result<T>` 与原始 GeoJSON 两种响应。

**验收标准**

- 生产构建中不存在 Mock 拦截代码；
- 页面请求能在后端访问日志中观察到；
- 资源、站点和空间查询均来自数据库；
- 后端返回 4xx/5xx 时，页面显示明确错误。

### P0-2：工单、派单、路径和状态迁移全部停留在 Pinia

**位置**

- `frontend/src/store/emergency.ts:111-199`
- `frontend/src/store/emergency.ts:143-165`
- `frontend/src/components/EmergencyDispatchPanel.vue:471-548`

**现象**

前端自行生成 `A_<timestamp>` 工单，直接修改 `待派单/处置中/已完成` 状态，并用四点折线模拟 OSRM。代码没有调用 `POST /api/work-orders`、`POST /api/work-orders/{id}/dispatch`、`PATCH /api/work-orders/{id}/status` 或 `/osrm/route/v1/**`。

**影响**

- 刷新页面后新增工单消失；
- 后端状态机、资源锁定/释放、状态历史和实时推送未被使用；
- 多用户同时操作时无法保持一致；
- 路径距离与时间来自前端估算，不能代表道路路由。

**建议**

1. 增加登录页、认证 Store 和带 Bearer Token 的请求客户端；
2. 工单列表以 `GET /api/work-orders` 为唯一来源；
3. 登记、派单和状态变更分别调用后端接口，并以响应对象刷新 Store；
4. 订阅 `/topic/work-orders` 处理其他客户端的变更；
5. 路径通过同源 `/osrm/route/v1/driving/**` 获取，数据库只保存服务端需要的距离、时间和资源分配信息；
6. 本地演示数据仅放在明确的 Mock 模式。

**验收标准**

- 刷新或更换浏览器后仍能查询同一工单；
- 非法状态迁移显示后端 `409` 原因；
- 同一泵车不能被两个工单同时派发；
- 完成/拒绝工单后，资源在后端恢复可用；
- 页面路线与 OSRM GeoJSON 一致。

### P0-3：公众上报仅写入 LocalStorage，未使用服务端追踪码

**位置**

- `frontend/src/components/MobileReportApp.vue:73-84`
- `frontend/src/components/MobileReportApp.vue:258-300`

**现象**

提交动作生成 `MR-<timestamp>`，把整条记录和图片 Data URL 写入浏览器 `localStorage`。页面没有调用 `POST /api/reports`、`POST /api/reports/{trackingCode}/images`、`GET /api/reports/track/{trackingCode}` 或 `GET /api/reports/nearby`。

**影响**

- 上报只在当前浏览器可见，后台无法受理；
- 图片可能迅速耗尽 `localStorage` 配额；
- 页面显示“上报成功”时，服务器没有任何记录；
- 用户无法跨设备追踪进度。

**建议**

1. 提交表单调用 `POST /api/reports`，以服务端 `trackingCode` 作为业务 ID；
2. 图片改为受控 Data URL 或 multipart 上传；
3. 将追踪码列表作为轻量本地历史，详情每次从服务端查询；
4. 地图附近积水点由 `GET /api/reports/nearby` 获取；
5. 离线场景应显示“待同步”，联网成功后再显示“上报成功”。

**验收标准**

- 提交成功后数据库存在对应 `work_order`；
- 页面展示服务端追踪码；
- 另一台设备可用追踪码查询状态；
- 图片能通过 `/uploads/**` 访问；
- 网络失败时不显示成功状态。

## P1：业务一致性与安全

### P1-1：最近泵车算法没有排除已派发车辆，演示种子状态也互相冲突

**位置**

- `frontend/src/store/emergency.ts:122-140`
- `frontend/src/store/emergency.ts:48-80`
- `frontend/src/mock/api.ts:61-84`

**现象**

`findNearestPump` 遍历所有泵车，没有检查 `status === '空闲'`。同时初始工单 A002 声明已占用 P002，而 Mock 中 P002 仍为“空闲”。

**影响**

同一车辆可能被重复推荐和派发；页面上的工单与资源状态无法互相证明。

**建议**

- 前端候选列表先过滤可用资源；
- 最终可派性由后端事务锁与资源状态校验决定；
- 删除互相矛盾的硬编码工单/资源，或由同一份种子数据生成；
- 后端返回 `409` 时刷新资源与工单列表。

**验收标准**

- 已派发泵车不出现在候选列表；
- 并发派单只有一个请求成功；
- 工单完成后候选列表自动恢复该资源。

### P1-2：空间绘制与警情登记同时监听右键，操作会相互触发

**位置**

- `frontend/src/components/SpatialQueryPanel.vue:388-392`
- `frontend/src/components/SpatialQueryPanel.vue:658-675`
- `frontend/src/components/EmergencyDispatchPanel.vue:460-469`
- `frontend/src/components/EmergencyDispatchPanel.vue:567-578`

**现象**

空间查询用右键结束多边形；应急调度用同一个 `contextmenu` 打开警情登记。两个监听器挂在同一 MapLibre 实例，缺少共享的交互模式和互斥判断。

**影响**

用户结束多边形时可能同时弹出警情登记框，坐标和模式状态容易混乱。

**建议**

- 建立全局地图交互模式，例如 `idle / draw-query / register-event`；
- 仅在 `idle` 或明确登记模式下响应警情右键；
- 绘制模块结束后显式恢复默认模式；
- 增加“绘制时右键只结束绘制”的组件测试或 E2E 用例。

**验收标准**

- 多边形绘制期间右键只结束绘制；
- 登记模式下右键只打开登记框；
- 关闭面板或组件卸载后不残留监听器。

### P1-3：多个 Popup 直接拼接未转义字段，接入真实数据后存在存储型 XSS

**位置**

- `frontend/src/components/MapContainer.vue:186-202`
- `frontend/src/components/EmergencyDispatchPanel.vue:173-182`
- `frontend/src/components/EmergencyDispatchPanel.vue:204-214`
- `frontend/src/components/EmergencyDispatchPanel.vue:250-260`
- `frontend/src/components/SpatialQueryPanel.vue:611-652`

**现象**

站点名称、地址、资源详情、报警描述和空间设施属性直接进入模板字符串，再传给 `Popup.setHTML`。这些字段将来可来自后台维护或公众输入。

**影响**

攻击者可把恶意 HTML 写入数据库；管理员或调度员打开地图弹窗时可能执行脚本。

**建议**

优先使用 DOM API 创建节点并通过 `textContent` 写入数据。确需富文本时，使用经过配置的 DOMPurify 白名单净化后再传入 `setHTML`。同时在 Nginx 增加适合项目资源来源的 Content-Security-Policy，作为第二层保护。

**验收标准**

- 名称或描述为 `<img src=x onerror=alert(1)>` 时只显示文本；
- 所有进入 `setHTML` 的动态字段均有统一安全处理；
- 自动化测试覆盖资源、警情和空间查询弹窗。

### P1-4：图片选择不限制大小和真实格式，也没有压缩

**位置**

- `frontend/src/components/MobileReportApp.vue:231-249`
- `frontend/src/components/MobileReportApp.vue:258-285`

**现象**

前端仅检查 `file.type.startsWith('image/')`，随后把原图读为 Data URL。没有限制 5 MB、没有限定 PNG/JPEG、没有压缩，也没有处理 `localStorage` 配额异常。

**影响**

- 与后端“PNG/JPEG、最大 5 MB、最多 2500 万像素”契约不一致；
- 移动端大图会占用大量内存和传输流量；
- 写入 `localStorage` 可能抛出配额异常并中断提交。

**建议**

- 选择时限制 `image/png`、`image/jpeg` 和 5 MB，并在压缩后校验像素与大小；
- 使用 Canvas 或成熟库将长边压到合理尺寸并控制质量；
- 上传前显示压缩后大小；
- 捕获读取、压缩、上传和本地存储异常；
- 以服务端最终校验结果为准。

**验收标准**

- 超限或不支持的图片在读取前被拒绝；
- 常见手机照片压缩后可稳定上传；
- 后端拒绝时页面保留表单并展示原因。

## P2：体验与工程质量

### P2-1：新手引导引用不存在的调度面板 ID

**位置**

- `frontend/src/components/HelperUI.vue:47-51`
- `frontend/src/components/EmergencyDispatchPanel.vue:642-649`

**现象**

引导目标为 `#emergency-dispatch-panel`，实际模板根节点只有 `class="gis-overlay-container"`，没有该 ID。引导逻辑找不到元素后会退回居中提示，无法高亮目标。

**建议与验收**

为稳定容器补充目标 ID，或改用现有唯一选择器。运行完整引导时，每一步都应有可见高亮区域。

### P2-2：地图销毁后 Store 仍保留旧 Map 实例引用

**位置**

- `frontend/src/components/MapContainer.vue:250-256`
- `frontend/src/store/map.ts:16-21`

**现象**

组件卸载时执行 `mapInstance.remove()` 并把局部变量设为 `null`，只更新 `isMapLoaded`，没有把 Pinia 中的 `mapInstance` 清空。

**影响**

重新挂载、页面模式切换或其他组件晚到的侦听器可能访问已销毁地图。

**建议与验收**

让 `setMapInstance` 接受 `Map | null`，卸载时同步清空 Store；反复进入主页面和移动上报页面时，不应出现“style is not done loading”或对已销毁地图的调用。

### P2-3：README 与页面元信息仍为 Vite 模板

**位置**

- `frontend/README.md:1-5`
- `frontend/index.html:1-8`

**现象**

README 仍是 Vue/Vite 脚手架说明，HTML 使用 `lang="en"` 和 `title=frontend`。

**建议与验收**

README 补充功能、运行方式、Mock 开关、后端契约和移动端入口；页面改为 `lang="zh-CN"` 并使用项目名称。浏览器标签、可访问性语言和仓库说明应与系统一致。

### P2-4：生产主包过大

**位置**

- `frontend/package.json:12-23`
- `frontend/src/App.vue:9-13`

**证据**

本轮 `npm run build` 成功，但 Vite 生成的主 JS 约 2.04 MB、gzip 后约 550.69 KB，并提示 chunk 过大。MapLibre、Turf、Naive UI 和全部业务面板在首屏一起进入主包。

**建议**

- 对移动上报页、空间查询面板和调度面板使用动态导入；
- 检查 Turf 是否可改为按函数导入；
- 配置 `manualChunks` 分离地图与 UI 依赖；
- 用构建分析器确认主要体积来源；
- 以真实设备首次加载时间和交互可用时间作为验收指标。

**验收标准**

首屏只加载当前页面所需模块；构建不再产生单个超大业务 chunk，低端移动设备进入上报页时无明显长时间白屏。

## 建议联调顺序

1. P0-1：让真实 API 请求到达后端；
2. 增加登录态、统一 API 客户端和错误展示；
3. P0-2：接通工单、派单、状态和 OSRM；
4. P0-3：接通公众上报、图片和追踪；
5. 接入 STOMP 监测、预警和工单主题；
6. 处理全部 P1；
7. 完成 P2 与端到端回归。

本文件仅提出修改意见，本轮没有执行上述前端改动。
