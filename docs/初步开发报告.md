## 摘要与阶段结论

本阶段已经形成一套可继续联调的后端与部署基线：Spring Boot 后端具备认证授权、PostGIS 空间查询、监测数据生成、预警流转、公众上报、工单调度、后台管理、操作审计和 STOMP/WebSocket 推送能力；Flyway 维护完整的数据模型；Compose 声明了 PostGIS、后端、OSRM 与 Nginx 前端四个服务。

当前证据支持以下结论：

- **已实现**：上述后端业务代码、数据库迁移、容器构建文件、反向代理和启动脚本已经写入仓库。
- **已自动验证**：原后端因控制器编译错误未能执行其 14 项测试；修复并补充安全、空间、附件、工单、监测和凭据测试后，当前 `mvn clean verify` 为 48/48 通过。前端 `npm run build` 通过，其中包含 TypeScript 类型检查；Nginx、Compose 配置解析与 ShellCheck 均通过。
- **已本地运行实测**：临时 PostgreSQL 16 + PostGIS 3.6 完成 Flyway V1—V3，Spring Boot 3.5.16 后端健康状态为 `UP`；关键 REST、权限、并发状态机、空间查询、公众上报、附件清理、图层上传、图片静态访问和 STOMP 权限已完成烟雾测试。
- **尚未完整容器实测**：当前宿主没有可用于整套服务冷启动的常驻 Podman/Docker 环境，因此 Compose 四服务、Nginx 同源代理、OSRM 下载与预处理、命名卷重启持久化仍未实际验证。
- **前端保持零修改**：本次没有修改 `frontend/**`。现有前端无条件加载 Mock，工单和公众上报仍保存在 Pinia 或 `localStorage`，会阻断真实后端联调。故当前成果可认定为“后端及部署代码基本完成，前端演示可构建”，不能认定八个模块均已端到端验收。

## 一、项目背景与建设目标

项目面向徐州市城市内涝场景，目标是以 WebGIS 为统一空间入口，连接监测站点、风险预警、公众上报、应急资源和处置工单。考虑到真实排水管网、传感器和应急资源数据难以获取，课程项目使用公开地理数据与可解释的模拟业务数据，重点验证空间检索、实时推送和业务状态流转。

技术路线保持全链路开源：

- 前端：Vue 3、TypeScript、MapLibre GL JS、Turf.js、Pinia、Naive UI；
- 后端：Java 17、Spring Boot 3.5.16、Spring Security、MyBatis-Plus、Spring WebSocket；
- 数据：PostgreSQL 15、PostGIS 3.4、Flyway；
- 路由：OSRM 5.27.1，采用 MLD 预处理；
- 部署：Compose、Nginx、多阶段容器镜像。

## 三、总体架构

系统按任务书中的五层结构实现，各层职责如下。

| 层次 | 当前组件 | 主要职责 |
|---|---|---|
| 前端交互层 | Vue 3、MapLibre GL JS、Turf.js、Pinia、Naive UI | 地图渲染、绘制与缓冲、列表联动、应急演示、移动端上报界面 |
| 后端业务层 | Spring MVC、Service、Spring Security、Scheduler、STOMP | REST 接口、参数校验、权限控制、监测模拟、预警和工单状态机、实时广播 |
| 数据存储层 | PostgreSQL/PostGIS、MyBatis-Plus、Flyway | 关系数据、空间几何、时序记录、状态历史、空间索引和版本化迁移 |
| 路径计算层 | OSRM | 基于 OSM 道路数据执行路径规划；当前已完成容器与同源代理声明，尚未运行 |
| 部署层 | Compose、Nginx、容器镜像、命名卷 | 四服务编排、健康检查、同源 API/WS/上传/OSRM 代理、数据持久化 |

后端内部继续采用 Controller、Service、Mapper、Entity/DTO 的分层组织。控制器处理 HTTP 契约，Service 负责事务与状态规则，Mapper 使用参数绑定访问数据库，Entity/DTO 分离持久化结构和输入边界。

## 四、数据模型与 PostGIS 设计

数据库由 `backend/src/main/resources/db/migration` 下的 V1—V3 版本化维护，启动时由 Flyway 自动执行。

### 4.1 核心表

| 领域 | 数据表 | 作用 |
|---|---|---|
| 用户与审计 | `sys_user`、`sys_role`、`sys_user_role`、`sys_config`、`sys_log` | 用户、角色、参数配置和操作日志 |
| 监测与预警 | `monitor_station`、`monitor_data`、`warning_record` | 空间站点、时序监测值、预警状态与影响范围 |
| 资源与工单 | `emergency_resource`、`work_order`、`work_order_status_history`、`work_order_attachment` | 应急资源、公众/预警/调度工单、流转历史与图片 |
| 空间图层 | `geo_layer`、`geo_feature`、`spatial_facility` | 上传图层元数据、通用 GeoJSON 要素和空间查询演示设施 |

### 4.2 空间与完整性策略

- 所有业务空间坐标采用 WGS84（EPSG:4326）。
- 站点、资源和工单保留 `lat/lng`，数据库触发器同步生成 `geometry(Point, 4326)`。
- 预警影响范围由 GeoJSON 文本同步为 PostGIS 几何。
- `monitor_station.geom`、`emergency_resource.geom`、`work_order.geom`、`geo_feature.geom`、`spatial_facility.geom` 和 `warning_record.affected_geom` 均配置 GiST 空间索引。
- 未处置公众上报另有 `geom::geography` 表达式 GiST 索引；处理中工单对 `assigned_resource_id` 有部分唯一索引，数据库层阻止同一资源重复占用。
- 类型、状态、坐标范围、数量、优先级和水深均有数据库 `CHECK` 约束；外键按业务需要采用级联删除或置空。
- 工单增加唯一追踪码、状态历史、附件、分配资源、路线距离/时间与关键处置时间戳。

该结构兼顾了课程演示和后续扩展。Flyway V1—V3 已在临时 PostgreSQL 16 + PostGIS 3.6 实际执行成功；Compose 目标镜像为 PostgreSQL 15 + PostGIS 3.4，其容器内冷启动仍需补测。

## 五、接口与实时通信

当前精确接口见 `docs/API_CONTRACT.md`。大部分业务接口统一返回：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

为兼容现有前端，以下接口有意直接返回原始数组或 GeoJSON：

- 无查询参数的 `GET /api/resources`；
- 无查询参数的 `GET /api/stations`；
- `POST /api/spatial-query`；
- `GET /api/reports/nearby`。

WebSocket 使用 SockJS/STOMP：

- 握手端点：`/ws`；
- 监测数据：`/topic/monitor`；
- 风险预警：`/topic/warning`；
- 工单变更：`/topic/work-orders`。

STOMP `CONNECT` 必须携带 `Authorization: Bearer <token>`。VIEWER 可订阅监测与预警，工单主题仅 ADMIN/OPERATOR 可订阅；客户端 `SEND` 一律拒绝，避免伪造广播。本轮已用原生 WebSocket/STOMP 和实际 `Origin` 连接 `/ws/websocket` 验证这些规则。业务消息在事务提交后发布，降低“消息已到达但数据库事务回滚”的不一致风险。

## 六、认证与安全设计

### 6.1 身份与权限

- 登录密码使用 BCrypt 保存和校验；
- JWT 默认有效期 24 小时，载荷包含用户 ID、用户名和角色；
- HTTP 会话采用无状态模式；
- 角色分为 `ROLE_ADMIN`、`ROLE_OPERATOR`、`ROLE_VIEWER`；
- 注册/建号接口仅管理员可用，新账号默认分配只读角色；工单列表及工单主题仅管理员和操作员访问；
- 用户、配置和日志接口仅管理员访问；
- 一般只读接口允许三类角色，新增/修改/删除允许管理员和操作员；
- 公众上报、追踪、附近查询以及现有地图所需的三个兼容接口开放匿名访问；
- 禁用用户时同时阻止“禁用当前账号”和“禁用最后一个可用管理员”。

### 6.2 输入与文件边界

- DTO 对长度、枚举、坐标、水深和分页范围进行 Bean Validation；
- 空间查询只接受 Polygon/MultiPolygon，并限制坐标深度、坐标数量和名称长度；
- MyBatis 空间查询使用 `#{...}` 参数绑定，未拼接外部输入；
- GeoJSON 图层限制类型、要素数、单文件大小和拓扑有效性；
- 公众图片只接受 PNG/JPEG，校验声明与实际格式、限制 5 MB、2500 万像素及每单 5 张；数据库失败或事务回滚会清理落盘文件；
- CORS 来源由环境变量配置；Nginx 增加基础安全响应头；
- 后端与 OSRM 容器使用非 root 用户并启用 `no-new-privileges`。

演示配置中仍保留默认数据库密码、默认账号和 JWT 密钥，且 CSRF 在无状态 API 模式下关闭。Nginx 已对登录、空间查询、匿名上报和图片追加限流；部署到共享网络前仍必须替换密钥、限制暴露端口、采用 HTTPS，并结合实际流量设置总量配额。

## 七、关键业务实现

### 7.1 空间查询

前端提交可选缓冲区 GeoJSON、名称和设施类型。后端先做 GeoJSON 结构与坐标校验，再以参数化 SQL 调用 PostGIS：

- 属性筛选：名称包含匹配、`waterlogging/pump` 类型匹配；
- 空间筛选：`ST_Intersects` 与 `ST_GeomFromGeoJSON`；
- 输出：标准 `FeatureCollection`，要素几何由 `ST_AsGeoJSON` 返回。

服务层与数据库查询边界已有测试覆盖；真实 PostGIS 烟测中，组合条件查询命中 4 个要素，非法几何返回 400。现有前端仍被 Mock 拦截，因此浏览器页面联调尚未完成。

### 7.2 监测与预警

定时任务默认每 5 秒为启用站点生成模拟值。水位、雨量和流量使用不同的基值、振幅和单位；水位与雨量按 `sys_config` 中的阈值计算 0—2 级状态。仅本批新数据会被广播和检测，避免停用站点的旧值反复触发；同一站点的预警创建按数据库行锁串行化并在 30 分钟内去重。

预警状态规则为：

```text
PENDING ──确认──> CONFIRMED ──发布──> PUBLISHED ──撤销──> REVOKED
   └──────驳回────────────────────────────────────> REJECTED
```

同时提供分页查询、状态统计和最长 366 天的时间范围回溯。

### 7.3 公众上报

匿名用户可提交经纬度、积水深度、描述、联系方式和可选 Data URL 图片。后端生成 `FR-...` 追踪码，并依据水深映射工单优先级。用户可按追踪码查看状态历史和图片，也可查询指定半径内未完成的公众上报 GeoJSON。

图片既可随 JSON 上报，也可通过独立 multipart 接口补充上传。追加上传锁定工单并限制类型、状态和数量；删除待处理/已拒绝工单时，附件记录与磁盘文件一并清理。当前移动端组件仍使用 `localStorage`，因此上述服务端流程尚未被现有页面消费。

### 7.4 工单与调度状态机

工单状态机限制为：

```text
PENDING ──派单或受理──> PROCESSING ──完成──> COMPLETED
   └────────拒绝────────> REJECTED
PROCESSING ──拒绝───────> REJECTED
```

派单仅允许待处理工单，资源必须为 `AVAILABLE` 且类型为泵车或救援队。事务内锁定工单与资源，数据库唯一索引兜底防止重复占用；普通资源 CRUD 不能伪造或释放 `DISPATCHED` 状态，也不能删除活动资源。工单完成或拒绝时释放资源；删除工单同样先加行锁，不能与派单竞态。每次迁移写入 `work_order_status_history`，并在事务提交后广播工单变化。

### 7.5 后台维护

后台接口覆盖用户分页与资料修改、账号启停、站点与资源 CRUD、阈值配置、审计日志、GeoJSON 上传和要素回读。图层上传支持 multipart 文件和原始 `application/json` 两种方式，统一落入 PostGIS。

## 八、部署设计

`compose.yaml` 声明四个服务：

1. `postgis`：PostGIS 15-3.4，业务数据使用命名卷；
2. `backend`：Java 17 运行时，依赖数据库健康检查，上传文件使用命名卷；
3. `osrm`：下载江苏 OSM PBF，执行 extract/partition/customize 后启动 MLD 路由；
4. `frontend`：多阶段构建 Vue，再由 Nginx 提供静态文件和同源代理。

Nginx 路由如下：

| 浏览器路径 | 上游 |
|---|---|
| `/api/**` | Spring Boot |
| `/ws/**` | Spring Boot WebSocket |
| `/uploads/**` | Spring Boot 上传目录 |
| `/osrm/**` | OSRM，代理时去掉 `/osrm` 前缀 |

默认宿主端口只绑定 `127.0.0.1`。PostGIS、后端、OSRM 和前端均定义健康检查；Nginx 配置匿名写入/登录/空间查询限流和安全响应头。OSRM 首次下载与预处理耗时较长，健康检查设置了较长启动窗口。

本次严格未修改 `frontend/**`。该约束保留了既有界面与交互成果，也留下了以下明确阻断：
