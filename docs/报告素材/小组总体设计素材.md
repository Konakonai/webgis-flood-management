# 小组总体设计报告素材

## 一、系统目标

系统面向徐州市城市内涝监测与应急处置，围绕“空间认知—风险发现—事件上报—资源调度—处置跟踪”形成业务闭环。系统同时服务调度人员、管理人员、只读查看人员和公众上报用户。

## 二、总体技术路线

- 表示层：Vue 3、TypeScript、Naive UI、Pinia。
- 地图与空间交互：MapLibre GL JS、Turf.js、GeoJSON。
- 统一接入：Nginx静态资源与同源反向代理。
- 业务层：Spring Boot、Spring Security、MyBatis-Plus、STOMP/WebSocket。
- 数据层：PostgreSQL、PostGIS、Flyway。
- 路由层：OSRM，使用江苏OpenStreetMap道路数据和MLD算法。
- 部署层：Podman/Docker Compose、健康检查与命名卷。

浏览器统一访问8088端口。Nginx将REST、WebSocket、附件和OSRM请求代理到相应容器，前端不直接感知内部服务地址。

## 三、功能组成

### 1. 地图工作台

提供底图浏览、主题切换、业务图层、属性弹窗、设施定位和全屏查看，作为空间查询与应急调度的共享地图容器。

### 2. 空间查询

支持矩形、多边形绘制、Turf缓冲、名称与设施类型过滤；后端通过PostGIS执行空间相交，结果以GeoJSON返回并与列表、标记联动。

### 3. 监测与预警

后端定时生成监测数据，依据配置阈值创建预警，提供确认、驳回、发布、撤销、统计和时间线接口，并通过STOMP推送变化。

### 4. 应急调度

展示工单、泵车驻地和物资点；调用OSRM计算道路路线；后端以事务和行锁完成派单、抵达、完成、拒绝和资源释放。

### 5. 公众上报

移动端采集位置、水深、描述和图片，服务端生成追踪码；公众可查询处理状态、历史和结果，业务人员可在工单系统继续处置。

### 6. 系统管理

提供用户与角色、站点、资源、空间图层、阈值配置和操作日志接口，使用JWT、RBAC和AOP审计保护管理操作。

## 四、核心数据设计

可在报告中将数据表分为五组：

1. 权限与配置：`sys_user`、`sys_role`、`sys_user_role`、`sys_config`、`sys_log`。
2. 监测与预警：`monitor_station`、`monitor_data`、`warning_record`。
3. 应急处置：`emergency_resource`、`work_order`、`work_order_status_history`、`work_order_attachment`。
4. 地理图层：`geo_layer`、`geo_feature`。
5. 联合查询设施：`spatial_facility`。

空间字段使用EPSG:4326并建立GiST索引。业务状态和坐标范围由CHECK约束保护，活动工单的资源分配由部分唯一索引保护。

## 五、关键业务流程

### 1. 空间查询流程

用户绘制范围 → 前端生成GeoJSON与可选缓冲区 → 组合名称和设施类型 → 调用空间查询API → PostGIS执行`ST_Intersects` → 返回FeatureCollection → 列表与地图同步展示。

### 2. 应急派单流程

选择待派单工单 → 获取空闲资源 → 请求OSRM比较道路路线 → 选择资源并提交派单 → 后端锁定工单与资源 → 同时更新工单和资源状态 → 提交事务 → STOMP广播刷新。

### 3. 公众上报流程

获取或选择位置 → 填写水深和描述 → 压缩并校验图片 → 服务端校验并创建工单 → 返回追踪码 → 公众按追踪码查询 → 业务端派单处置 → 返回状态历史与结果。

### 4. 预警流程

定时生成监测值 → 保存监测数据 → 阈值检测 → 创建待确认预警 → 人工确认或驳回 → 已确认预警发布 → 必要时撤销 → 各状态变化实时广播。

## 六、前后端协作接口

- 地图状态与图层由`useMap`向各业务组件提供。
- 全局主题由`useTheme`统一管理。
- 空间查询通过`POST /api/spatial-query`交换GeoJSON。
- 资源与站点通过`/api/resources`、`/api/stations`交换数据。
- 工单通过`/api/work-orders`完成创建、派单和状态迁移。
- 认证通过`/api/auth/login`签发JWT。
- 实时变化通过`/topic/monitor`、`/topic/warning`和`/topic/work-orders`推送。
- 道路路线经Nginx的`/osrm`代理访问OSRM。

## 七、安全与可靠性设计

- BCrypt密码、JWT与三级RBAC。
- 请求参数、GeoJSON、分页、枚举、坐标和图片均设置边界。
- 动态地图弹窗统一HTML转义。
- 工单派发、状态迁移、图片追加和管理员启停使用事务或行锁。
- 事务提交后再发送WebSocket通知。
- Nginx配置登录、空间查询和匿名上报限流，以及CSP、nosniff、frame和referrer安全头。
- 后端与OSRM使用非root用户，宿主端口默认绑定127.0.0.1。

## 八、部署结构与运行结果

Compose包含PostGIS、backend、osrm、frontend四个服务。PostGIS负责空间与业务数据，backend提供REST和STOMP，OSRM提供道路路径，frontend容器中的Nginx提供页面和统一代理。

完成空卷冷启动后，Flyway依次执行V1—V4，OSRM完成江苏数据预处理，四个容器均为healthy。浏览器通过统一入口完成登录、全图空间查询、最近泵车路线、派单、抵达、完成、资源释放、公众上报和追踪查询。

## 九、测试证据

- 后端：50项自动化测试全部通过。
- 前端：TypeScript检查与Vite生产构建通过。
- 服务配置：Compose解析和Nginx配置检查通过。
- 端到端：空间查询返回8个设施；OSRM返回约6.9千米、约9分钟的实际道路路线；工单完成后两个泵车均恢复空闲；公众上报返回服务端追踪码并可查询到PENDING状态。

## 十、报告中可强调的设计特点

1. 前端空间计算与后端PostGIS查询结合：前端即时反馈，后端负责持久数据检索。
2. OSRM路线与工单状态机结合：路线展示不仅是地图效果，还服务真实派单事务。
3. 公众上报与内部工单共用处置模型：减少两套业务记录之间的转换成本。
4. 事务锁、数据库约束和实时消息共同保证多客户端状态一致。
5. 四服务容器编排使数据库、业务、路由和前端能够一次启动并重复验收。
