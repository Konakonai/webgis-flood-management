# 插图缺口与制作方案

## 一、参考文档中的图片类型

“杭州智能旅游服务系统总体设计报告”包含6幅主要图片，承担的不是界面展示，而是解释系统设计：

1. 系统功能模块图：树状展示系统、一级模块和二级功能。
2. 三层体系结构图：浏览器、Web/WebGIS服务、数据库之间的关系。
3. 用户操作或请求处理流程图。
4. GIS服务请求处理流程图。
5. 系统硬件结构框图：客户端、交换设备、服务器和网络连接。
6. 系统软件结构总体框图：子系统、空间数据服务和数据库关系。

其可借鉴之处是“每个重要章节都有一幅能够解释结构或流程的图”。旧式电脑、服务器剪贴画和带聊天水印的截图不应照搬，建议统一重绘为简洁的黑白或灰阶矢量风格。

## 二、当前已有图片

目前已有两类可用界面截图：

- 桌面端地图工作台全景，能够展示左右面板、地图、空间对象和调度资源。
- 窄屏底部抽屉界面，能够展示移动/窄窗口下地图与空间分析面板的共存方式。

这两张适合写入同学A个人报告，但不足以支撑总体设计报告。总体设计报告仍缺少结构图、流程图、数据图，以及多个关键业务结果截图。

## 三、建议用Python绘制的图片

下列图片应由Python脚本统一绘制。推荐使用`matplotlib`绘制版式，使用`networkx`计算关系图布局，使用`matplotlib.patches`绘制圆角矩形、箭头和分组框；导出300 dpi PNG和SVG两个版本。颜色以黑、深灰、浅灰为主，可用一种低饱和蓝色区分外部服务，但不使用渐变和剪贴画。

### P1 系统功能模块图（必须）

层级建议：

- 徐州市城市内涝应急指挥平台
  - 地图工作台：底图、图层、定位、属性查看
  - 空间查询：绘制、缓冲、属性过滤、结果联动
  - 监测预警：实时监测、阈值检测、预警流转、时间线
  - 应急调度：资源、工单、OSRM路线、状态跟踪
  - 公众上报：定位、图片、追踪码、进度查询
  - 系统管理：用户角色、站点资源、图层、配置日志

用途：总体设计“功能需求”或“系统功能结构”章节。

### P2 系统总体架构图（必须）

从上到下绘制：用户角色 → Vue/MapLibre表示层 → Nginx统一接入 → Spring Boot业务服务与OSRM → PostgreSQL/PostGIS与文件/路由数据卷。旁边标注STOMP、REST、GeoJSON和HTTP路线请求。

用途：总体设计“总体设计”章节。它可以替代参考文档中过时的浏览器—WebGIS服务器—数据库三层剪贴画。

### P3 Compose部署拓扑图（必须）

绘制浏览器8088、frontend/Nginx、backend:8080、postgis:5432、osrm:5000，以及`/api`、`/ws`、`/uploads`、`/osrm`四条代理路径和三个命名卷。

用途：总体设计“运行环境/部署设计”与同学A Podman工作说明。

### P4 空间查询时序或流程图（建议）

绘制“绘制范围—Turf缓冲—提交GeoJSON—ST_Intersects—返回FeatureCollection—列表与地图联动”。可以用横向六步流程，不必画复杂泳道。

用途：同学A个人报告“具体工作”和总体设计“空间查询流程”。

### P5 应急派单流程图（必须）

绘制“选择工单—筛选空闲资源—OSRM计算—提交派单—事务锁定—更新双状态—事务提交—STOMP广播”，并在资源不可用分支返回冲突。

用途：总体设计“业务流程”和前后端协作说明。

### P6 工单状态机图（建议）

主要状态：PENDING → PROCESSING → COMPLETED；PENDING/PROCESSING可进入REJECTED；抵达现场作为PROCESSING内部事件；完成或拒绝触发资源释放。

用途：同学B个人报告“工单业务设计”。

### P7 预警状态机图（建议）

状态：PENDING → CONFIRMED → PUBLISHED → REVOKED；PENDING → REJECTED。标注监测阈值触发PENDING，人工操作驱动后续状态。

用途：同学B个人报告“风险预警流程”。

### P8 核心数据ER图（必须）

只画核心关系，避免把所有字段塞入一页：

- `monitor_station` 1—N `monitor_data`
- `monitor_station` 1—N `warning_record`
- `emergency_resource` 1—N `work_order`
- `work_order` 1—N `work_order_status_history`
- `work_order` 1—N `work_order_attachment`
- `geo_layer` 1—N `geo_feature`
- `sys_user` N—M `sys_role`

用途：总体设计“数据结构设计”。

### P9 接口与实时消息关系图（可选）

绘制前端各组件与REST端点、STOMP主题、OSRM之间的消费关系，可用组件—接口两列矩阵图表达。

用途：总体设计“功能需求与程序关系”。若正文已有清晰表格，可不画。

## 四、需要实际运行系统后补拍的图片

这些图片必须来自真实运行页面或终端，不适合Python模拟：

### S1 登录与角色状态

画面要求：登录框或登录成功后的用户角色区域，能看出管理员身份。不要展示密码。

### S2 空间绘制与缓冲结果

画面要求：地图上同时可见绘制边界、缓冲面、查询结果点和结果列表。建议使用多边形而不是全图查询，以体现空间分析过程。

### S3 OSRM调度路线

画面要求：选中的待派单工单、候选泵车、地图道路路线、距离与预计时间同时可见。这是同学A个人报告最重要的功能截图之一。

### S4 工单状态变化

画面要求：同一工单处于“处置中/已抵达/已完成”之一，资源状态与工单状态能够相互对应。可与S3分开拍摄。

### S5 公众移动上报

画面要求：窄屏表单包含定位、水深、描述和图片入口；避免使用无内容的空表单。

### S6 追踪码与处理进度

画面要求：提交成功后的`FR-...`追踪码、当前状态和状态历史。对电话号码等个人数据做遮挡。

### S7 Swagger/OpenAPI或系统管理接口

画面要求：Swagger按认证、监测、预警、工单、空间图层、系统管理分组展开，或展示一个成功的分页接口响应。用于证明同学B完成统一后端接口。

### S8 预警接口或实时消息

优先方案：浏览器或API工具展示预警列表、统计和状态变化；若要证明实时推送，可拍摄WebSocket/STOMP收到`/topic/monitor`或`/topic/warning`消息的开发者工具画面。

### S9 Podman四服务健康状态

画面要求：终端执行`podman ps`或`podman-compose ps`，清晰显示postgis、backend、osrm、frontend均为healthy；命令提示符中不要出现无关私人路径。

### S10 自动化测试结果

画面要求：Maven输出50项测试、0 failure、0 error，以及前端生产构建成功。建议分别截图，保证文字可读。

## 五、个人报告与小组报告的最小图片组合

### 同学A个人报告

建议使用5幅：P4空间查询流程、S2绘制缓冲、S3调度路线、S5或S6公众上报、P3或S9 Podman部署。

### 同学B个人报告

建议使用5幅：P8核心ER图、P5派单流程或P6状态机、P7预警状态机、S7 Swagger、S10后端测试。

### 小组总体设计报告

建议使用8至10幅：P1、P2、P3、P5、P8，加上桌面全景、S2、S3、S5/S6和S9。正文中一幅图只承担一个解释目标，不重复放置相近截图。

## 六、Python绘图输出规范

- 画布：优先16:9横图或A4正文可容纳的1.5:1横图。
- 字体：中文使用思源黑体或Noto Sans CJK SC，英文和数字使用Times New Roman或Liberation Serif。
- 线条：主轮廓1.2—1.5 pt，内部关系0.8—1.0 pt。
- 字号：标题14—16 pt，节点10—11 pt，注释8—9 pt。
- 输出：SVG用于Word清晰缩放，PNG使用300 dpi作为兼容版本。
- 命名：`图2-1_系统总体架构.svg`、`图3-2_应急派单流程.svg`等，编号在定稿时统一。
- 图内不写大段说明，技术解释放在图题后的正文中。
