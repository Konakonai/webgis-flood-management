# WebGIS 内涝监测与应急管理系统——最终答辩技术问答指南

本指南已按最终代码和验收范围更新。答辩时应优先说明真实实现和可验证证据，不再把项目早期设想作为已完成功能。

## 问题 1：系统采用了什么总体架构？

**回答要点：**

系统采用前后端分离的 B/S 架构。Vue 3 与 MapLibre GL JS 负责地图和业务交互，Turf.js 在浏览器完成绘制与缓冲区计算；Spring Boot 提供认证、空间查询、监测预警、工单和公众上报接口；PostgreSQL/PostGIS 保存关系和空间数据；OSRM 使用江苏 OSM 道路数据计算路线。最终通过 Compose 编排 PostGIS、backend、osrm、frontend 四个服务，由 Nginx 在 8088 端口统一代理 REST、STOMP、附件和路由请求。

## 问题 2：为什么最终没有使用天地图？

**回答要点：**

项目早期试验过天地图，但 Key、域名、请求稳定性和调用配额会影响答辩环境。最终日间使用 OSM 瓦片，夜间使用 Carto 暗色瓦片，二者均采用标准 Web Mercator。GPS、PostGIS、GeoJSON 和 OSRM 统一使用 WGS84，从源头避免 GCJ-02/BD-09 纠偏问题，也提高了离线部署之外的整体可重复性。

## 问题 3：没有真实传感器和市政数据，监测功能如何验证？

**回答要点：**

静态设施由 Flyway 迁移脚本生成具有徐州空间语义的演示数据；动态水位和雨量由 Spring Boot 定时任务根据波动模型生成并写入数据库。系统默认每 5 秒向 `/topic/monitor` 推送新值，随后由 `WarningDetector` 根据配置阈值创建预警。这样能够验证数据存储、阈值判断、预警状态机和实时通信；未来接入真实设备时只需替换数据输入源。

## 问题 4：空间查询为什么同时使用 Turf.js 和 PostGIS？

**回答要点：**

两者职责不同。Turf.js 用于浏览器端的矩形/多边形绘制、缓冲区生成和即时视觉反馈；PostGIS 用于权威数据检索。前端将 GeoJSON 和属性条件提交到 `/api/spatial-query`，后端校验几何并执行参数化 `ST_Intersects`，再返回 FeatureCollection。这样既保证交互速度，也保证查询结果来自数据库。

## 问题 5：OSRM 路径规划是不是模拟连线？

**回答要点：**

不是。前端从空闲泵车中筛选候选资源后，通过 Nginx 的 `/osrm` 代理请求本地 OSRM。OSRM 基于江苏 OSM 道路网返回 GeoJSON 路线、距离和预计时间；前端再用 MapLibre 绘制路线及流光粒子动画。路线展示服务于真实派单事务，而不是只画起终点直线。

## 问题 6：如何防止同一辆泵车被重复派发？

**回答要点：**

后端在派单事务中对工单和资源执行 `SELECT ... FOR UPDATE`，校验双方状态后同时更新；数据库还通过部分唯一索引限制同一资源出现在多个活动工单中。事务提交后才向 `/topic/work-orders` 广播。发生冲突时整个事务回滚，因此不会出现工单已更新但资源未更新的中间状态。

## 问题 7：公众上报的图片和进度如何处理？

**回答要点：**

移动端支持 GPS 和地图中心选点。图片先在浏览器通过 Canvas 将最长边缩放至 1920 像素以内，再上传到后端附件接口。服务端校验文件类型、大小和像素数，将文件保存到持久化卷，并生成 `FR-...` 追踪码。公众可以在其他设备输入追踪码查询工单状态和处理历史。

## 问题 8：为什么图层解析放在后端？

**回答要点：**

初期考虑过在浏览器使用 `shapefile.js`，最终改为后端统一接收 GeoJSON 或 ZIP。后端集中检查文件大小、GeoJSON 结构、坐标维度、闭合环、拓扑有效性、要素和坐标数量，并在事务中写入 PostGIS。这比由不同浏览器分别解析更容易保证安全限制、错误语义和数据一致性。

## 问题 9：双人开发如何降低联调风险？

**回答要点：**

项目先约定 REST、GeoJSON、STOMP 和 OSRM 的接口边界。前端通过 `useMap()`、`useTheme()` 和统一 `apiRequest()` 隔离地图、主题和网络访问；后端通过 SpringDoc OpenAPI 和统一 `Result` 响应维护契约。最终联调时所有浏览器请求都走 Nginx 同源入口，避免开发地址、跨域和 WebSocket 路径不一致。

## 问题 10：项目做了哪些安全和可靠性处理？

**回答要点：**

- Spring Security、BCrypt、JWT 和三级角色权限。
- STOMP CONNECT 阶段校验 Bearer Token，限制客户端只能订阅授权主题。
- 数据库参数化空间 SQL、事务行锁、唯一约束和 Flyway 迁移。
- 图片类型、大小、像素和存储路径校验，失败时清理物理文件。
- Nginx 登录/查询/匿名上报限流以及 CSP、nosniff、frame、referrer 安全头。
- 容器非 root 运行、`no-new-privileges`、健康检查、持久化卷和自动重启。

## 问题 11：哪些功能是最终范围之外的？

**回答要点：**

项目没有交付天地图、地名搜索、距离/面积量测、ECharts 态势大屏、声音报警、PDF 导出、预警地图逐帧回放和完整后台管理页面。这些属于早期扩展设想。最终项目将双人开发时间集中在空间查询、公众上报、OSRM 调度、工单并发一致性、后端监测预警和容器化部署，形成了可以实际运行和重复验收的核心闭环。

## 问题 12：如何证明系统不是只有界面？

**回答要点：**

可以现场展示以下证据：

1. 登录后执行矩形或多边形查询，后端返回 PostGIS FeatureCollection。
2. 选择工单后由 OSRM 返回真实道路路线，再完成派单、抵达和完成。
3. 工单完成后泵车自动恢复空闲，并由 STOMP 刷新客户端。
4. 手机端提交图片后获得服务端追踪码，并能再次查询状态。
5. `podman compose ps` 显示四个服务，数据库迁移和持久化卷可重复启动。
6. 自动化测试、前端构建、Compose 和 Nginx 配置检查均可重复执行。
