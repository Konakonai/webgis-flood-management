## 🙋 常见问题及高分回答策略

### 问题 1：请介绍一下你们系统的总体架构设计？
*   **💡 核心答题思路**：从“前端交互层、后端业务层、数据存储层、路径计算层、部署层”五维回答，强调**100%开源生态的自洽性**。
*   **🗣 答辩回答模板**：
    > “各位老师好，我们的系统采用的是**全链路开源的 WebGIS 架构**。
    > *   **前端交互层**：基于 `Vue 3`（组合式 API）作为基础框架，采用 `MapLibre GL JS` 作为核心 WebGL 地图渲染引擎，配合 `Turf.js` 执行前端的空间拓扑几何计算（如缓冲区判定和量测），UI 组件库选用 `Naive UI` 以保证风格统一和极速响应。
    > *   **后端业务层**：基于 `Spring Boot 3` 框架，利用其内嵌的轻量级定时任务进行监测数据模拟，并采用 `WebSocket` 技术实现监测数值的实时主动广播；接口文档通过 `SpringDoc OpenAPI` 自动生成。
    > *   **数据存储层**：采用标准的开源空间数据库 `PostgreSQL + PostGIS` 扩展，用于存储行政区划、易涝点等多边形和点线要素，并支持空间索引。
    > *   **路径计算层**：本地容器化部署开源路由引擎 `OSRM`，导入徐州市 OSM 道路网，为应急泵车和物资匹配提供毫秒级的路径分析。
    > *   **部署层**：使用 `Podman Compose` 进行声明式多容器全栈部署，保证环境的一致性与迁移便捷性。”

### 问题 2：你们为什么不用高德/百度地图 API 做底图和路径规划，而是自己折腾这一套开源技术栈？
*   **💡 核心答题思路**：从**学术深度、配额限制、坐标偏移、代码侵入性**四个维度进行降维打击。
*   **🗣 答辩回答模板**：
    > “主要基于三个方面的考虑：
    > 1.  **坐标系一致性（规避 GCJ-02 纠偏）**：国内商业 API（高德/百度）强制使用加密的 GCJ-02 或 BD-09 坐标系，若将 GPS 原生数据或天地图数据叠加其上，必须编写复杂的纠偏算法（如 coordtransform），这会产生额外的计算开销且引入误差。而我们这套开源技术栈，从底图（OSM）、路径引擎（OSRM）、空间数据库（PostGIS）到前端渲染，**全链路原生使用标准 WGS84 坐标系**，从源头上彻底消除了国内火星坐标系带来的偏移问题。
    > 2.  **没有配额限制**：国内商业 API 均有严格的每日调用配额限制。我们本地化部署 OSRM 开源引擎，数据源全部本地化，**完全免费且无任何请求配额限制**，适合进行高频的大规模资源路径分析。
    > 3.  **技术自主可控**：MapLibre GL JS 支持现代 GPU 硬件加速和高度自定义的矢量瓦片样式渲染，相比商业闭源 SDK，我们可以完全控制图层的生命周期、着色器交互和路径流光动画，技术实现深度更符合 GIS 专业的学术要求。”

### 问题 3：不同地图底图和业务数据叠加时，如果坐标系不一致（如天地图使用 CGCS2000），你们怎么保证地图叠加不发生偏移？
*   **💡 核心答题思路**：区分**地理坐标系**和**投影坐标系**，说明通过“底图墨卡托对齐 + 前端重投影”解决。
*   **🗣 答辩回答模板**：
    > “这是一个典型的 WebGIS 多源数据融合问题。我们通过以下两步解决：
    > 1.  **底图层对齐**：天地图国内官方服务虽然是 CGCS2000（EPSG:4490），但在非高精测绘场景下它与 WGS84 的形变偏差在毫米级，可以直接融合。更重要的是，我们**直接接入天地图官方提供的 Web 墨卡托投影（EPSG:3857）瓦片服务**，从而与 OpenStreetMap 底图以及 MapLibre GL JS 的世界投影基准线直接对齐，确保底图层之间零偏差。
    > 2.  **数据层重投影**：我们的业务要素（物资、积水点）在 PostGIS 后端以 `WGS84 (EPSG:4326)` 形式存储。当通过 GeoJSON 传入前端时，MapLibre GL JS 引擎会在加载图层时，**利用 GPU 硬件加速在前端自动完成向 Web 墨卡托（EPSG:3857）的重投影计算**，实现与底图的无缝贴合。”

### 问题 4：作为学生实习项目，你们没有真实的内涝设备监控数据（如电子水尺水位），也没有真实的城市管网保密数据，怎么做可行性验证？
*   **💡 核心答题思路**：阐述 **“Mock（模拟测试）”** 的软件工程思想，说明管网通过典型 GeoJSON 模拟，动态监测通过“模拟发生器 + WebSocket”机制实现。
*   **🗣 答辩回答模板**：
    > “在工业界软件工程中，面对未就绪或涉密的硬件接口，使用 **Mock（数据模拟）** 是标准的可行性验证方案。我们的具体策略是：
    > 1.  **静态资产（管网/泵站）**：我们根据徐州市主干道路拓扑，使用 GIS 软件绘制了一套具有典型几何网络特征的模拟城市雨水管线 GeoJSON 要素，用于在 PostGIS 数据库中进行拓扑演示。
    > 2.  **动态数据（实时水位/雨量）**：我们在 Spring Boot 后端设计了一个**‘历史时序数据发生器’**，利用正弦波结合随机波动的数学模型，按合理物理逻辑定时产生水位、雨量变化。然后利用 **WebSocket 双向主动通信通道**，向前端看板大屏和地图 Marker 实时推送数据流。这套机制能够完美验证系统的实时告警、专题图刷新和大屏渲染能力，未来当有真实 IoT 设备接入时，仅需更换数据输入源即可。”

### 问题 5：后台管理里有一个‘空间图层在线上传’功能，Shapefile 是二进制多文件格式，你们在后端如何解析？
*   **💡 核心答题思路**：说明**“前端解析，一元化 API 传输”**的轻量化方案，减轻后端负担。
*   **🗣 答辩回答模板**：
    > “传统方案是在后端使用臃肿的 GeoTools 库或依赖 GDAL 动态库，配置复杂且易内存泄漏。
    > 我们的创新做法是**‘计算下移，前端解析’**：当管理员上传打包的 Shapefile 文件时，我们在前端利用 `shapefile.js` 库在浏览器中直接解压解析二进制流，将其转换为标准的 `GeoJSON` 对象。随后通过常规的 HTTP POST 请求，将解析好的 GeoJSON 发送给 Spring Boot，后端只需要将其作为 JSON 反序列化，并直接使用 PostGIS 的 `ST_GeomFromGeoJSON` 存入数据库。这种做法将二进制解析工作彻底从后端剥离，极大地简化了系统架构。”

### 问题 6：双人合作开发，你们怎么保证接口能无缝对接，不出现互相等待的情况？
*   **💡 核心答题思路**：介绍 **“契约优先开发 (Contract-First)”** 理念，以及基于 SpringDoc OpenAPI 的协作机制。
*   **🗣 答辩回答模板**：
    > “我们制定了严格的**‘契约优先’**协作流程：
    > 1.  在第 0 周，我们共同设计了系统的 8 个协作接口（IF-1 至 IF-8），并使用标准格式（TypeScript 接口类型及 Java DTO 对象）进行了精确定义。
    > 2.  后端引入了 `SpringDoc OpenAPI`。前两周后端框架搭建完毕后，即使业务代码未写完，后端也能直接暴露出标准的 Swagger/OpenAPI 描述文档。
    > 3.  前端（同学A）可以直接根据 OpenAPI 定义进行契约式 Mock 开发，封装跨组件共享的 `useMap()` 和 `useTheme()` 组合函数；后端（同学B）则依据契约填充实际的 SQL 和业务逻辑。第 7 周联调时，由于双方一直遵循同一份强类型契约，可以实现一插即用、零阻碍对接。”
