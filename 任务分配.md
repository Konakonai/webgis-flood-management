# WebGIS 内涝监测与应急管理系统——协作任务分配方案

为保障学生实习项目的顺利高效开展，本项目采用双人分工协作的模式进行开发。在保证系统架构先进合理的基础上，充分解耦前后端，按“技术主线”将 8 大功能模块划分为两条开发轨道。

---

## 一、 核心分配逻辑

系统开发划分为以下两条核心轨道：
*   **轨道A（GIS 前端专家）**：围绕地图引擎与前端空间交互（`MapLibre GL JS` + `Turf.js`）展开。主要负责构建核心地图底座、图层可视化控制、前端空间计算、移动端定位交互以及本地 OSRM 路径规划动画渲染。
*   **轨道B（全栈业务专家）**：围绕后端服务与核心业务流转（`Spring Boot 3` + `PostgreSQL/PostGIS` + `ECharts`）展开。主要负责数据库设计、时序数据 WebSocket 定时推送、后端空间查询检索、应急工单审批流程以及基于 Podman Compose 的全栈服务部署。

### 👥 最终分配结果

| 负责角色 | 负责模块 | 核心技术栈 | 工作量估算 | 主要职责 |
| :--- | :--- | :--- | :---: | :--- |
| **同学A**<br>(GIS前端专家) | **模块一**：地图基础浏览<br>**模块二**：空间查询检索<br>**模块五**：应急调度管理<br>**模块八**：通用辅助交互 | Vue 3 + MapLibre GL JS<br>+ Turf.js + Pinia | ~4.5 周 | 负责封装核心地图管理器 `useMap()` 供全局调用，实现空间测距、缓冲区计算及前端渲染、OSRM 路径动画。 |
| **同学B**<br>(全栈业务专家) | **模块三**：监测大屏与看板<br>**模块四**：风险预警发布<br>**模块六**：公众定位上报<br>**模块七**：系统配置与后台 | Spring Boot 3 + PostGIS<br>+ WebSocket + Naive UI | ~4.5 周 | 负责设计 PostgreSQL 表结构与空间表、编写 API、提供时序数据发生器及 WebSocket 服务，并使用 Naive UI 开发后台管理与大屏。 |

## 二、 同学A：GIS前端专家 任务详情

### 2.1 负责模块与技术要点

#### 模块一：地图基础浏览与可视化交互（核心 WebGIS 底座）
*   **子功能开发**：地图平移缩放、天地图墨卡托投影瓦片（EPSG:3857）与 OSM 底图加载与切换、图层显隐与透明度滑块控制、基于 Turf.js 的量测工具（距离与面积）、基于天地图 API/Nominatim 的地理搜索定位、基于 MapLibre Popup 的自定义 HTML 信息弹窗。
*   **🔧 关键技术产出**：封装 Vue 3 组合式函数 `useMap()`。该 composable 作为系统的核心地图管理器，向全局暴露 MapLibre 地图实例、图层动态注册接口以及地图交互事件钩子。

#### 模块二：空间查询与检索（前端空间计算）
*   **子功能开发**：框选与多边形绘制交互、利用 Turf.js 在前端进行实时缓冲区多边形计算并在地图渲染、发起空间+属性的联合检索条件表单、在地图上联动闪烁高亮展示过滤出的要素。
*   **🔗 接口依赖**：调用同学B提供的空间检索 REST API，上传查询几何（GeoJSON）与属性过滤条件。

#### 模块五：应急调度与资源管理（路径规划交互）
*   **子功能开发**：利用 MapLibre Marker 及 Symbol 图层可视化渲染防汛物资点、泵车驻地；地图右键打点报警登记表单；调用本地部署的 OSRM 路由引擎 API，将返回的折线解码为 GeoJSON 路径并在地图上实现流光箭头动画；工单调度指令发送与状态追踪。
*   **🔗 接口依赖**：调用同学B提供的应急资源及工单管理 CRUD API；路径导航则向 OSRM 发起异步路由请求并自行完成前端渲染。

#### 模块八：通用辅助交互功能
*   **子功能开发**：用户操作指南与新手引导步骤遮罩（使用 Naive UI 或轻量向导库）、全局亮色与暗色模式主题切换（结合 CSS 变量及 Naive UI 主题上下文）、全屏显示控制。
*   **🔧 关键技术产出**：封装全局主题管理器 `useTheme()` 暴露给同学B的大屏看板与上报页面，实现地图样式与 UI 组件颜色的深浅同步切换。

## 三、 同学B：全栈业务专家 任务详情

### 3.1 负责模块与技术要点

#### 模块七：后台数据维护与系统管理（全栈业务基石）
*   **子功能开发**：基于 Spring Security + JWT 的用户登录与细粒度权限控制；监测站点/应急物资的信息表维护；空间地理图层在线上传，**在前端利用 `shapefile.js` 快速解析 Shapefile 二进制包为 GeoJSON，并通过 REST API 发送至后端存入 PostGIS**；系统参数配置；通过 Spring AOP 统一记录审计日志。
*   **🔧 关键技术产出**：基于 SpringDoc OpenAPI 规范，自动生成并维护系统统一的 RESTful 接口体系与文档。

#### 模块三：监测数据实时展示与统计分析（态势大屏与通信）
*   **子功能开发**：采用 CSS Grid / Flexbox 开发响应式态势感知大屏；使用 ECharts 绘制水位/雨量历史时间序列折线图与多站点对比柱状图；水位超限时触发前端声音与图标红色高亮报警；大屏组件报告导出（使用 `jsPDF` + `html2canvas` 导出为 PDF）。
*   **🔧 核心技术产出**：搭建 WebSocket 服务端；在后端启动轻量定时任务（`@Scheduled`），使用数学波动模型自动生成水位与降雨模拟时序数据，通过 WebSocket 向大屏和地图广播。

#### 模块四：风险预警与发布流程
*   **子功能开发**：气象/积水风险告警人工复核与审批工作流；预警发布记录与状态流转；历史预警回溯与播放时间轴（后端按时间范围查询受灾 GeoJSON 多边形，前端通过时间滑块配合 MapLibre 逐帧渲染）。
*   **🔗 接口依赖**：调用同学A提供的 `useMap()` 接口将回溯的灾情范围图层动态加载至地图实例中。

#### 模块六：公众上报与信息反馈（移动端/公众端）
*   **子功能开发**：移动端自适应上报页面；一键定位（HTML5 Geolocation API）；积水表单与现场图片压缩上传；附近积水点避让查询（调用前端 Turf.js 进行缓冲区判定，检索未处置积水点）；上报进度追踪列表。
*   **🔗 接口依赖**：使用同学A的 `useMap()` 实例化微缩地图进行定位展示，使用 `useTheme()` 适配手机端深浅色风格。

## 四、 🔗 接口契约清单 (IF-1 ~ IF-8)

| 契约编号 | 接口名称 | 提供方 | 消费方 | 说明与技术协议 |
| :---: | :--- | :---: | :---: | :--- |
| **IF-1** | `useMap()` | 同学A | 同学B | 前端地图生命周期与实例共享 Composable，暴露底图加载、飞行定位及标记管理接口。 |
| **IF-2** | `useTheme()` | 同学A | 同学B | 全局 UI 主题管理器，支持一键切换系统明/暗色，同时触发 MapLibre 地图底图样式（Light/Dark）联动。 |
| **IF-3** | 空间查询 API | 同学B | 同学A | `POST /api/spatial-query`<br>入参：空间几何（GeoJSON 多边形）与检索条件；出参：空间要素集合（GeoJSON FeatureCollection）。 |
| **IF-4** | 监测与资源 API | 同学B | 同学A | `GET/POST/PUT/DELETE /api/stations` 与 `/api/resources`<br>提供监测站点及防汛物资点属性及空间坐标的常规持久化接口。 |
| **IF-5** | 工单系统 API | 同学B | 全局 | `GET/POST/PUT/DELETE /api/work-orders`<br>支持事件上报、下发处置、派单、状态反馈等状态机流转。 |
| **IF-6** | 用户认证 (JWT) | 同学B | 全局 | `POST /api/auth/login` 与路由守卫机制，返回 Token 并解析角色身份。 |
| **IF-7** | 实时推送通道 | 同学B | 全局 | 基于 WebSocket 连接，广播格式为 `{ stationId, value, timestamp, type: "rain" }` 的实时时序流。 |
| **IF-8** | 图层注册接口 | 同学A | 同学B | 挂载于地图管理器上的方法：`registerLayer(id, layerOpts)` 与 `removeLayer(id)`，供大屏渲染专题图使用。 |

## 五、 📅 协作开发时序建议

```
第 0 周：脚手架与环境初始化（共同完成）
├── 前端：Vue 3 + Vite + TypeScript 初始化，集成 Naive UI 与 Pinia 状态管理
├── 后端：Spring Boot 3 项目初始化，集成 SpringDoc OpenAPI，配置 PostgreSQL/PostGIS 链接
└── 部署：编写本地 Podman Compose 配置文件（一键拉起 PostGIS 及 OSRM 路由基础容器）
```

```
第 1 - 2 周：核心底座并行开发
├── 同学A：构建核心 WebGIS 地图底座，加载天地图 Web 墨卡托底图，产出 useMap() 容器 (IF-1)
└── 同学B：开发模块七后台系统管理，提供系统用户认证 (IF-6) 及 stations/resources API 基础 (IF-4)
```

```
第 3 - 4 周：空间交互与大屏开发
├── 同学A：开发模块二空间查询，对接前端 Turf.js 缓冲区计算，调用 IF-3 空间检索
└── 同学B：开发模块三数据大屏，建立 ECharts 图表，搭建 WebSocket 实时推送逻辑 (IF-7)
```

```
第 5 - 6 周：业务流转与外围功能
├── 同学A：开发模块五应急调度，集成 OSRM 引擎实现路径计算，编写前端流光箭头动画
└── 同学B：开发模块四预警发布流及模块六移动端公众上报（通过 IF-1 获取地图定位）
```

```
第 7 - 8 周：联调测试与项目整合
├── 全局联调：验证 WebSocket 推送频率、OSRM 路径稳定性以及空间图层解析入库 (shp.js)
├── 主题对齐：联合调试 useTheme() 明暗色主题，保证大屏和地图夜间模式完美自适应
└── 部署交付：编写一键 Podman Compose 部署脚本，准备答辩演示数据与项目说明文档
```

## 六、 📊 分配依据总结

*   **技术隔离性高**：同学A专精地图渲染与前端算法，产出的 `useMap()` 与主题样式向外暴露，同学B无需关心底层的 WebGL/WebGPU 坐标变换与地图渲染生命周期。
*   **降低集成风险**：前期定义好了 IF-1 至 IF-8 契约及标准的 TypeScript 接口，接口定义直接映射至 SpringDoc OpenAPI。开发过程中，双方可通过标准 Mock 服务独立调试，确保第 7 周联调阶段一插即用。
*   **工作量均衡**：同学 A 在 GIS 前端方向需要应对较多前端拓扑运算与 OSRM 路线渲染动画，学习调研成本略高；同学 B 则需要承担全栈业务开发，代码编写量较大，两者整体工作量在 8 周的工期内非常均衡。
