# WebGIS 内涝监测与应急管理系统——最终任务分工与实施变更记录

本文档由项目初期任务分配方案更新而来，以 2026 年 7 月最终提交的代码、接口契约、测试结果和容器化部署为准。项目开发过程中根据演示稳定性、双人工作量和核心业务闭环进行了范围调整；早期设想但未进入最终交付的功能不再作为验收项。

## 一、最终目标与分工原则

系统围绕“空间认知—公众上报—资源调度—处置跟踪”形成可运行闭环，并提供监测预警、权限审计和数据维护等后端能力。两名成员按技术主线协作：

- **同学 A（前端 GIS 与部署）**：负责 Vue 3/MapLibre 地图工作台、Turf.js 空间交互、OSRM 路线表达、应急调度前端、移动公众上报、主题与响应式交互，以及 Podman Compose 整机部署。
- **同学 B（后端业务与数据）**：负责 Spring Boot、Spring Security/JWT、PostgreSQL/PostGIS、空间查询、监测预警、工单状态机、公众上报与附件、系统管理接口、WebSocket/STOMP 和自动化测试。

## 二、同学 A：前端 GIS 与 Podman 部署

### 2.1 地图工作台

- 使用 MapLibre GL JS 加载 OSM 日间瓦片和 Carto 暗色瓦片；主题切换时同步更新地图与界面颜色。
- 展示城区范围、排水管网、监测点、积水点、泵车、物资、查询结果和路线等业务对象。
- 封装 `useMap()`，共享地图实例并提供 `registerLayer()`、`removeLayer()`、显隐和透明度等图层管理能力。
- 使用安全转义后的 MapLibre Popup 展示属性，提供缩放、平移、飞行定位和 Marker 联动。

### 2.2 空间查询

- 实现矩形和多边形绘制，并处理绘制模式与右键灾情登记之间的事件所有权。
- 使用 Turf.js 按用户设置的距离生成缓冲区并在地图上分层渲染。
- 调用 `POST /api/spatial-query` 执行空间与属性联合检索。
- 将返回的 GeoJSON 与列表、Marker、飞行定位和闪烁高亮联动。

### 2.3 应急调度

- 展示泵车、物资点和工单位置，并支持右键登记事件。
- 从空闲泵车中筛选候选资源，调用同源 `/osrm/route/v1/driving/...` 接口获取真实道路路线、距离和预计耗时。
- 使用 MapLibre GeoJSON 图层绘制路线及流光粒子动画。
- 调用工单创建、派单、抵达和完成接口，并订阅 `/topic/work-orders` 刷新工单与资源状态。

### 2.4 移动公众上报

- 提供独立窄屏页面，支持 HTML5 Geolocation 和地图中心选点。
- 使用 Canvas 将现场图片最长边限制为 1920 像素后上传。
- 调用 `/api/reports` 创建公众工单、上传附件，并保存服务端生成的 `FR-...` 追踪码。
- 支持凭追踪码重新查询状态和处置历史。

### 2.5 通用交互与部署

- 实现登录入口、新手引导、明暗主题、全屏控制和窄屏工作区切换。
- 使用 Compose 编排 PostGIS、Spring Boot、OSRM 和 Nginx 四个服务。
- 配置健康检查、命名卷、非 root 运行、同源反向代理、请求限流和 CSP 等安全响应头。

## 三、同学 B：后端业务与数据服务

### 3.1 认证、权限与系统管理

- 使用 Spring Security、BCrypt 和 JWT 实现登录认证。
- 设置 ADMIN、OPERATOR、VIEWER 三级权限，并对 REST 与 STOMP 订阅执行鉴权。
- 提供用户、监测站点、应急资源、系统配置、操作日志和空间图层接口。
- 使用 Spring AOP 与 `@LogOperation` 记录关键业务操作。
- 使用 SpringDoc OpenAPI 维护接口文档。

### 3.2 PostGIS 与空间服务

- 使用 Flyway V1—V4 管理业务表、空间表、约束、索引和演示数据。
- 业务几何统一采用 WGS84/EPSG:4326，并建立 GiST 空间索引。
- 使用参数化 `ST_Intersects` 完成空间与属性联合检索。
- 支持 GeoJSON 或 ZIP 图层上传；几何解析、合法性检查和 PostGIS 入库由后端统一完成。

### 3.3 监测与预警

- 后端定时生成水位/雨量模拟数据，保存数据库并默认每 5 秒向 `/topic/monitor` 推送。
- 根据系统配置阈值自动创建预警。
- 提供 PENDING、CONFIRMED、REJECTED、PUBLISHED、REVOKED 状态流转、统计和时间线接口。
- 预警变化通过 `/topic/warning` 广播。

### 3.4 工单与公众上报

- 统一管理内部工单和公众上报工单，记录状态、位置、指派资源、附件和状态历史。
- 工单支持创建、派单、抵达、完成、拒绝和资源释放。
- 使用数据库行锁、唯一约束和事务提交后广播保证并发一致性。
- 为公众上报生成追踪码，提供图片上传、进度查询和附近未处置上报的 GeoJSON 查询接口。

### 3.5 测试与质量保障

- 覆盖 Security、WebSocket、种子账号、监测数据、预警、工单、附件、资源、图层、空间查询和用户保护等自动化测试。
- CI 执行后端测试、前端构建、Compose 配置解析、Nginx 检查、ShellCheck 和 actionlint。

## 五、实施过程中的主要范围变更

| 初期设想 | 最终处理 | 原因 |
| :--- | :--- | :--- |
| 天地图与 OSM 多底图 | 改为 OSM 日间 + Carto 暗色底图 | 避免 Key、域名和配额影响演示稳定性 |
| 地名搜索、距离/面积量测、图层滑块面板 | 未纳入最终交付 | 聚焦空间查询与应急调度核心闭环 |
| ECharts 态势大屏、声音报警和 PDF 导出 | 调整为后端监测、阈值预警和实时接口 | 双人项目优先保证数据、状态机和部署完整性 |
| 前端 `shapefile.js` 解析 | 改为后端统一解析 GeoJSON/ZIP | 集中执行几何校验、事务入库和安全限制 |
| 预警地图逐帧回放 | 保留后端时间线 API，未提供前端播放界面 | 将预警管理作为可扩展后台能力 |
| 移动端附近避让界面 | 保留后端 `/api/reports/nearby` GeoJSON 接口 | 当前页面聚焦上报、附件和进度追踪 |

## 六、最终验收范围

最终演示以以下链路为准：登录与角色识别 → 空间绘制及 PostGIS 联合查询 → 公众定位上报和图片上传 → 调度员选择工单与空闲泵车 → OSRM 计算道路路线 → 派单、抵达、完成与资源释放 → STOMP 状态刷新 → 公众凭追踪码查询处理进度。部署侧验证四服务冷启动、健康检查、Flyway 迁移、Nginx 同源代理和持久化卷。
