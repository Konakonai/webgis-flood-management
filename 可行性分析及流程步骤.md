# WebGIS 内涝监测与应急管理系统——实施可行性与最终流程

本文档由项目初期可行性设想更新而来，以最终代码和实际验收结果为依据。系统面向徐州市城区内涝应急场景，在缺少真实 IoT 设备和专有市政数据的条件下，采用公开 OSM 道路数据、模拟监测数据和自建业务数据完成可重复演示。

## 一、最终技术路线

- **前端**：Vue 3、TypeScript、MapLibre GL JS、Turf.js、Pinia、Naive UI。
- **后端**：Spring Boot 3、Spring Security、MyBatis-Plus、SpringDoc OpenAPI、STOMP/WebSocket。
- **数据库**：PostgreSQL 15、PostGIS 3.4、Flyway V1—V4。
- **路径引擎**：OSRM 5.27，使用江苏 OpenStreetMap 道路数据和 MLD 算法。
- **部署**：Podman/Docker Compose，包含 PostGIS、backend、osrm、frontend 四个服务；Nginx 作为统一入口。

## 三、核心业务流程

### 3.1 空间联合查询

1. 用户选择矩形或多边形绘制模式。
2. 前端生成 GeoJSON 几何，并使用 Turf.js 计算可选缓冲区。
3. 前端调用 `POST /api/spatial-query` 提交几何、设施名称和类型条件。
4. 后端校验几何，并使用参数化 `ST_Intersects` 查询空间设施。
5. 前端将 FeatureCollection 渲染为列表和 Marker，并提供飞行定位与闪烁高亮。

### 3.2 应急派单

1. 调度员选择待处理工单。
2. 前端读取空闲泵车并以直线距离筛选候选资源。
3. 前端通过 `/osrm` 代理请求道路路线，获得 GeoJSON、距离和预计时间。
4. 用户确认派单后，后端锁定工单与资源并校验状态。
5. 事务成功后更新双方状态，并向 `/topic/work-orders` 广播。
6. 抵达、完成或拒绝继续写入状态历史；完成或拒绝时释放资源。

### 3.3 公众上报

1. 公众通过手机定位或地图中心选择积水位置。
2. 浏览器使用 Canvas 将图片最长边缩放至 1920 像素以内。
3. 页面调用 `/api/reports` 创建公众工单，再上传图片附件。
4. 后端生成 `FR-...` 追踪码并保存状态历史。
5. 公众可在当前或其他设备输入追踪码查询处置状态。

### 3.4 监测与预警

1. 后端定时任务生成水位/雨量模拟值并写入数据库。
2. 默认每 5 秒向 `/topic/monitor` 推送新值。
3. `WarningDetector` 根据系统阈值创建待确认预警。
4. 预警支持确认、驳回、发布和撤销，并通过 `/topic/warning` 广播。
5. 统计、历史和时间线通过 REST API 提供给后续管理界面。

## 四、数据来源与模拟策略

- **底图与道路**：OSM/Carto 提供地图瓦片；OSRM 使用江苏 OSM 道路数据。
- **基础设施**：数据库迁移脚本提供具有徐州空间语义的监测点、泵车、物资和空间设施演示数据。
- **动态监测**：后端使用带波动的数学模型生成水位/雨量数据，用于验证存储、阈值检测和实时推送。
- **公众数据**：通过真实页面提交到统一工单模型，图片保存到持久化附件卷。

## 五、坐标与空间一致性

- GPS、业务几何、PostGIS 和 OSRM 均使用 WGS84/EPSG:4326。
- MapLibre 将 GeoJSON 自动投影到 Web Mercator 画布显示。
- 项目不使用高德、百度等 GCJ-02/BD-09 数据源，避免额外纠偏误差。
- 日间 OSM 与夜间 Carto 均采用标准 Web Mercator 瓦片，不依赖天地图 Key 或调用配额。

## 六、图层上传方案

最终方案由后端统一接收 GeoJSON 或 ZIP 图层文件，完成文件限制、结构解析、坐标维度、闭合环、拓扑有效性、要素数量和坐标数量校验，并在单一事务中写入图层及其要素。该方案替代了初期“浏览器使用 `shapefile.js` 解析”的设想，使安全限制、错误语义和事务回滚集中在服务端。

## 七、部署可行性

Compose 包含以下四个服务：

1. `postgis`：保存空间与业务数据，使用健康检查和持久化卷。
2. `backend`：提供 REST、STOMP、附件和 Flyway 迁移。
3. `osrm`：下载并预处理江苏道路数据，提供路线服务。
4. `frontend`：Nginx 分发静态页面并代理 `/api`、`/ws`、`/uploads` 和 `/osrm`。

所有宿主端口默认绑定 `127.0.0.1`；服务配置 `restart: unless-stopped`、`no-new-privileges`、健康检查和依赖顺序。Nginx 对登录、空间查询、公众上报和图片上传设置限流，并返回 CSP、nosniff、frame 和 referrer 等安全响应头。

## 八、范围调整说明

初期设想中的天地图、地名搜索、距离/面积量测、ECharts 态势大屏、声音报警、PDF 导出、前端 Shapefile 解析、预警地图逐帧回放和移动端附近避让界面未进入最终交付。这些功能不影响登录、空间查询、上报、调度、路线、处置、追踪及部署主链路，最终验收不再将其列为已完成功能。

## 九、最终验收结果

- 前端 TypeScript 检查和 Vite 生产构建通过。
- 后端 Security、WebSocket、预警、工单、附件、图层、监测和空间查询等自动化测试通过。
- Compose 配置与 Nginx 配置检查通过。
- rootless Podman 完成四服务冷启动，容器达到健康状态。
- OSRM 返回真实道路路线。
- 浏览器经 8088 统一入口完成登录、空间查询、工单派发、抵达、完成、资源释放、公众上报和追踪查询闭环。
