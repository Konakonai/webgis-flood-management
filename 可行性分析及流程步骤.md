# WebGIS 内涝监测与应急管理系统——可行性验证报告

本项目旨在面向徐州市开发一个 WebGIS 内涝监测与应急管理系统。作为学生实习项目，在无专业实验设备的情况下，系统所依赖的所有底图与空间业务数据均通过网络公开资源及模拟生成技术获取。

本系统采用 100% 开源的先进技术方案：
*   **前端**：Vue 3 + MapLibre GL JS + Turf.js（空间几何计算）+ ECharts（数据大屏）+ Naive UI（前端组件库）
*   **后端**：Spring Boot 3 + PostgreSQL/PostGIS（空间数据库）+ SpringDoc OpenAPI（自动接口文档）
*   **路径引擎**：OSRM（开源路由引擎，基于 OSM 道路网数据本地化部署）
*   **容器编排**：Podman Compose（声明式多容器全栈编排，用于一键本地运行 PostGIS、Spring Boot 与 OSRM）

### 1.9 数据获取与模拟综合结论
本系统在底图和空间计算等核心 GIS 能力上完全使用成熟的开源免费服务，无需任何商业付费。针对排水设施、管网和实时监测设备等保密或不可获取的数据，采用以下合理的模拟机制：
1. **静态设施（管网/泵站）**：根据徐州市主要道路结构，手动生成部分 GeoJSON 模拟数据作为演示图层。
2. **动态监测（水位/雨量）**：编写后端定时任务生成器，按合理物理逻辑（如降雨正弦波动模型）每隔一定时间自动生成水位，通过 **WebSocket** 广播更新。
3. **路径规划（OSRM）**：本地部署 OSRM，以纯 WGS84 路线数据参与所有应急匹配，规避商业 API 配额和国内坐标偏移。

## 三、关键技术障碍与解决方案

### 3.1 障碍一：多源空间底图坐标系偏移
*   **问题根源**：天地图国内默认数据多为 CGCS2000（EPSG:4490）；OSM 矢量与影像瓦片使用 Web 墨卡托投影（EPSG:3857）；OSRM 与 GPS 终端均基于原生 WGS84（EPSG:4326）；若直接混合使用，因参考椭球体和投影模型不同，底图叠加时会产生数百米偏移。
*   **解决方案**：
    1.  **从源头规避 GCJ-02**：项目全面采用开源底图（OSM）与开源路由引擎（OSRM），在数据采集（WGS84）、路径分析（WGS84）、空间数据库存储（PostGIS 中的 `EPSG:4326`）到前端渲染中，不引入国内任何加密坐标系，从而彻底规避 GCJ-02 纠偏算法。
    2.  **底图统一为墨卡托投影**：天地图底图服务一律请求其官方提供的 **Web墨卡托投影（EPSG:3857）** 瓦片，从而与 OpenStreetMap 底图以及 MapLibre GL JS 的世界投影基准天然对齐。
    3.  **前端重投影机制**：当 MapLibre 渲染来自 PostGIS 的业务 GeoJSON 矢量图层（WGS84）时，引擎会自动在 GPU 中将其投影转换为 EPSG:3857 进行空间贴合，开发者无需手动介入重投影计算。

### 3.2 障碍二：天地图 API 日调用配额限制
*   **问题根源**：天地图个人开发者账号在每日底图瓦片调用和地名搜索 API 上有调用额度限制。
*   **解决方案**：
    1.  **开启前端本地缓存**：MapLibre GL JS 原生通过 HTTP Header 对底图瓦片开启 IndexedDB 或浏览器强缓存，避免重复请求相同的瓦片。
    2.  **配置开发者 Key 轮询池**：在前端封装请求拦截器，内置 2-3 个注册的开发者 Key，当某 Key 达到频次或超额错误时自动轮换。实际上对于学生开发与答辩演示，个人开发者配额已完全绰绰有余。

### 3.3 障碍三：二进制 Shapefile 文件的后端解析复杂度
*   **问题根源**：Shapefile 格式由多个二进制文件组成。传统后端使用 GeoTools 依赖库繁重，配置复杂，易发生字符集乱码及内存泄露。
*   **解决方案**：
    *   **前端解析方案**：在后台管理页面中，上传压缩的 Shapefile 包。前端利用 `shapefile.js` 依赖库直接在浏览器解析二进制流，将几何信息与 dbf 属性组合生成标准的 JSON/GeoJSON 对象。
    *   **一元化 API 传输**：前端解析完成后，将清洗后的标准 GeoJSON 数据通过常规的 RESTful API 发送给后端 Spring Boot。后端只需反序列化并使用 PostGIS 的 `ST_GeomFromGeoJSON` 快速入库，将复杂度彻底从后端物理层剥离。

### 3.4 障碍四：多容器全栈服务的本地一键化部署
*   **问题根源**：项目包括了 PostGIS 数据库、Java 后端、Nginx 静态代理以及 OSRM 路由服务，学生在本地逐个配置依赖环境和网络非常困难。
*   **解决方案**：
    *   基于 `Podman Compose` 编写全栈编排声明文件，配置容器间的网卡互通。
    *   使用 `podman-compose up -d` 即可在本地无特权环境下（符合 NixOS 用户无根运行要求）一键拉起整套系统。
